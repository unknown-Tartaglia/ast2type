# ast2type 核心模块

本目录包含JavaScript类型推导工具的核心实现模块。

## 模块概览

### 1. fact.ts - 事实系统

**职责**：定义和存储类型推导过程中的事实（Facts）。

**核心概念**：
- `Fact`：程序中的类型相关操作，如变量声明、函数调用、运算等
- `FactStore`：事实存储容器，管理所有收集到的事实
- `VarId`：变量的唯一标识符
- `TypeId`：类型的唯一标识符

**事实类型**：
- 流事实（FlowFact）：变量赋值和传递
- 操作事实（UnaryOpFact, BinaryOpFact, TernaryOpFact）：一元、二元、三元运算
- 对象事实（AllocObjFact, PropReadFact, PropWriteFact）：对象创建和属性访问
- 函数事实（AllocFunctionFact, CallFact）：函数定义和调用
- 其他事实：条件表达式、类型断言等

### 2. nType.ts - 类型系统

**职责**：定义类型结构和类型操作。

**核心类型**：
```typescript
type nType =
  | { kind: "primitive"; name: string }                    // 基础类型：number, string, boolean等
  | { kind: "literal"; value: string | number | boolean } // 字面量类型：1, "hello", true
  | { kind: "array"; elementType: number }                // 数组类型
  | { kind: "function"; name: string; ... }               // 函数类型
  | { kind: "union"; types: number[] }                    // 联合类型
  | { kind: "object"; name: string; properties: ... }     // 对象类型
  | { kind: "enum"; name: string; members: ... };         // 枚举类型
```

**核心组件**：
- `tNodeStore`：全局类型存储，管理TypeId到nType的映射
- 类型操作：创建、查询、修改、合并、序列化

### 3. rule.ts - 规则系统

**职责**：将JavaScript语言语义编码为类型推导规则。

**核心组件**：
- `Rule`：规则接口，定义规则的匹配和应用逻辑
- `RuleStore`：规则存储和分发，按事实类型索引规则
- `GraphEffect`：规则应用产生的图效果（添加边、合并节点等）

**规则分类**：
- **传播规则**：处理变量赋值和传递
- **操作规则**：处理各种运算操作
- **对象规则**：处理对象创建、属性访问
- **函数规则**：处理函数定义、调用
- **控制流规则**：处理条件、循环等

### 4. graph.ts - 类型图

**职责**：构建和维护类型约束图。

**核心概念**：
- `TypeGraph`：类型约束图，节点表示类型变量，边表示类型约束
- `NodeState`：节点状态，包含当前类型和属性信息
- `Edge`：类型关系边，表示子类型关系

**图操作**：
- 添加/移除节点和边
- 类型传播
- 节点合并
- 冲突检测

### 5. solver.ts - 求解器

**职责**：执行类型约束求解。

**核心算法**：
1. **初始化**：应用所有规则构建初始约束图
2. **迭代求解**：使用工作列表算法进行类型传播
3. **收敛检测**：检查类型是否稳定或达到最大迭代次数

**求解流程**：
```
事实收集 → 规则应用 → 约束图构建 → 工作列表迭代 → 类型收敛 → 结果输出
```

### 6. strategy.ts - 策略模式

**职责**：定义类型推导策略，支持不同的求解策略。

**当前实现**：
- `DeterminantStrategy`：确定性策略，基于固定规则进行类型传播
- 计划支持概率化策略，引入不确定性进行更灵活的推导

### 7. meta.ts - 元数据存储

**职责**：存储AST节点的元信息，为类型推导提供上下文。

**存储内容**：
- 位置信息：文件路径、行列号、字符偏移
- 语义信息：变量名、函数名、类名、属性名
- 绑定关系：参数映射、作用域链

## 数据流

### 类型推导流程

```
源代码 → AST → 事实收集 → 规则应用 → 约束图构建 → 类型求解 → 类型输出
```

### 模块间交互

1. **入口模块** (`ast2type.ts`) 调用 `code2ast.ts` 生成AST
2. **事实系统** 从AST中提取类型相关事实
3. **规则系统** 将事实转换为类型约束
4. **类型图** 构建约束图并管理节点状态
5. **求解器** 执行迭代求解直到收敛
6. **类型系统** 提供类型创建和操作支持
7. **元数据** 为整个过程提供上下文信息

## 扩展指南

### 添加新规则

1. 在 `fact.ts` 中定义新的事实类型
2. 在 `rule.ts` 中实现对应的规则类
3. 在 `RuleStore` 中注册新规则
4. 测试规则的正确性

### 支持新类型

1. 在 `nType.ts` 的 `nType` 类型定义中添加新类型
2. 在 `tNodeStore` 中添加序列化和打印支持
3. 实现处理新类型所需的规则
4. 更新类型合并策略

## 测试

每个模块都有对应的测试用例，位于 `tests/` 目录中。测试覆盖：

1. 基础类型推导
2. 复杂类型结构
3. 规则应用正确性
4. 求解器收敛性

## 性能考虑

- 类型图使用高效的数据结构存储节点和边
- 工作列表算法只处理发生变化的节点
- 类型合并使用哈希去重避免重复计算
- 大图处理采用增量更新策略
